ความคิดเห็น และแผนการของผม

บริบท: เว็บลูกค้าเก่า 10 ปี ไม่มีเอกสาร ไม่มี test คนเก่าลาออกหมด มีเวลา 2 สัปดาห์ทำ discovery ไม่ต้องแก้จบ

ผมไม่เคยเจอ legacy system แบบนี้ตรงๆมาก่อน เคยแต่นึกภาพว่าถ้าเจอจะเป็นยังไง เพราะงั้นแนวทางของผมจะเน้นความระมัดระวังเป็นหลักก่อน ไม่รีบ

คำถามที่ 1: 3 วันแรก

Day 1: ศึกษาข้อมูล อ่านจาก docs อื่นๆของงานที่ใกล้เคียงกันเท่าที่มีให้ แยกประเภท จำแนกข้อมูลก่อนว่าอะไรกระทบ prod หรือมีความเสี่ยงมาก เสี่ยงน้อย พร้อม backup ข้อมูลเพื่อไม่ให้เกิดความเสี่ยงต่อข้อมูล และผู้ใช้งานที่ใช้อยู่

Day 2: ใช้ AI ในการช่วยหา อ่าน สรุป ให้ว่างานคืออะไร มีอะไรที่สามารถจับได้โดยไม่เสี่ยง ไม่กระทบต่อ prod หรือ user พร้อมจัดทำบันทึกเพื่อบันทึกการทำงาน และลำดับขั้นตอนอย่างมีระบบ เพื่อให้สามารถทำงานเป็น step และเดินหน้าต่อไปได้ถูกทาง

Day 3: พอรู้ควาเสี่ยง backup ข้อมูลแล้ว debug ปัญหา รายงานผลเพื่อหาแนวทางการแก้ไขที่ถูกต้องในเรื่องที่ลูกค้าต้องการ

ใช้ AI ยังไง (ตัวอย่าง prompt ที่จะลองใช้จริง):
- "นี่คือ error log จาก server เก่า [แปะ log] ช่วยดูให้หน่อยว่ามี pattern อะไรที่บ่งชี้ว่าเว็บช้าเพราะอะไรได้บ้าง"
- "ช่วย list checklist สิ่งที่ควรเช็คก่อนแตะ production server ที่ไม่มีเอกสารเลย เพื่อไม่ให้พังระหว่างสำรวจ"
- "ถ้าต้องการ deploy feature ใหม่ สามารถทำได้อย่างไร เพื่อไม่ให้เกิดผลกระทบต่อลูกค้าที่ใช้งานอยู่ใน server"
- "ข้อแนวทางการแก้ไขปัญหา ทั้งหมดอย่างไรทำเป็น docs เพื่อให้สามารถอ่านควบคู่กับการทำงานได้อย่างถูกต้อง"

ถ้าไม่รู้จะถามใคร:
- พี่ในทีมที่ส่งงานมาให้ - เรื่อง context ธุรกิจ ประวัติปัญหาเก่าๆ
- sysadmin ที่ดูแล infra - เรื่องสิทธิ์เข้าถึง, deploy process เดิมทำยังไง
- ถามกลุ่ม หรือ platform ที่เจอเหตุการณ์ใกล้เคียงกัน
- ถาม AI

เรื่องที่กังวล: กลัวว่าแค่สำรวจเฉยๆก็ดันไป trigger อะไรที่กระทบ prod เพราะไม่เคยเจอระบบแบบนี้ตรงๆมาก่อน เลยอยากทำทุกอย่างที่เป็น read-only ให้มากที่สุดในช่วงแรก ค่อยขยับไปทำอะไรที่เสี่ยงขึ้นทีหลัง

จบ 3 วัน
- Discovery docs สั้นๆ (architecture เท่าที่เข้าใจ, root cause เบื้องต้นของความช้า, risk list, คำถามที่ยังไม่มีคำตอบ)
- เสนอลำดับความสำคัญของงานสัปดาห์ถัดไป

คำถามที่ 2: ไฟล์ .php obfuscate ใน images/
คิดว่าคืออะไร: ไม่แน่ใจเลยจริงๆครับ อาจเป็นไฟล์ที่หลงเหลือจากการทำงานเก่า หรืออาจไม่สำคัญอะไร แต่อย่าเพิ่งลบน่าจะดีที่สุดครับ มีส่วน code ที่ถูกบังไว้อาจเพราะมีเหตุผลกันเรื่องความปลอดภัย

คำถามก่อนตัดสินใจ:
- มี log ไหมว่าไฟล์นี้เคยถูกเรียกใช้งานจริงบ้าง ล่าสุดเมื่อไหร่
- ระบบหลังได้ใช้งานไฟล์นั้นบ้างไหม อย่างไร

จะทำยังไง: หลังจากศึกษา ไฟล์ .php ที่อยู่ในโฟลเดอร์ images/ มักเป็นไฟล์สำหรับ จัดการรูปภาพ (เช่น อัปโหลด ลบ หรือปรับขนาด) หรือเป็นโค้ดที่แฮกเกอร์ฝังไว้ (Malware) หากคุณไม่ได้เป็นคนสร้างไฟล์นั้นเอง

ยังไม่ลบ เพราะถ้าเป็นไฟล์สำหรับ จัดการรูปภาพ อาจมีผลต่อ prod

หรือถ้าเป็น Malware ที่แฮกเกอร์แอบอัปโหลดเข้ามาไว้บนเว็บเซิร์ฟเวอร์จริงการลบทันทีจะเสียหลักฐานที่ใช้สืบต่อได้ จะกักไฟล์ไว้ก่อน (ย้ายออกจาก path ที่ execute ได้ หรือ backup แยกต่างหาก) และไม่รันไฟล์นี้บน prod เด็ดขาด

อะไรต้องเช็คเพิ่ม:
- server log ว่ามีการเข้าถึงไฟล์นี้จาก IP แปลกๆไหม
- ไฟล์อื่นในโฟลเดอร์เดียวกันที่ modify time ใกล้เคียงกัน
- สิทธิ์ของโฟลเดอร์ images/ ว่าอนุญาตให้ execute PHP ได้จริงไหม (ถ้าใช่ นั่นคือสัญญาณอันตราย)

ใครควรรู้: พี่ในทีมที่ส่งงานมาให้ก่อน แล้วถ้าข้อมูลชี้ไปทางเรื่องความปลอดภัยจริง ต้องแจ้งขึ้นไปอีกระดับ (security/lead) และอาจต้องแจ้งลูกค้าด้วย เพราะกระทบความปลอดภัยของระบบที่ยังมีคนใช้งานอยู่จริง
