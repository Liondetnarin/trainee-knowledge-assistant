## Outline พร้อมคำถามช่วยคิด (map ไปที่ template.md)

### 1. ภาพรวม & หลักการ (~100–150 คำ)
- คุณเชื่ออะไรเรื่องการ grow คน? (feedback ถี่? autonomy vs structure? psychological safety?)
- อยากให้ผู้ตรวจรู้สึกว่านี่คือ "ปรัชญาที่คุณเชื่อจริง" ไม่ใช่ท่องมาตอบ

### 2. สัปดาห์ที่ 1 — First Impression
- Day 1: อะไรที่ทำให้ trainee ไม่กลัว/ไม่หลง? (codebase tour, dev env setup, ใครคือ buddy)
- Day 2–3: pairing กับใคร ทำ task เล็กแค่ไหน
- Day 4–5: เป้าหมายสัปดาห์แรกคืออะไร (เช่น local env รันได้ + เข้าใจ architecture คร่าว ๆ)

### 3. แผนรายสัปดาห์ (สัปดาห์ 2–16)
ตัวอย่างการแบ่งช่วง (ปรับตามที่คุณคิดว่าจริง):
- สัปดาห์ 2: setup + shadow senior
- สัปดาห์ 3–4: first PR เล็ก ๆ ต้อง merge ได้
- สัปดาห์ 5–8: เริ่มรับ feature ของตัวเอง (มี code review ถี่)
- สัปดาห์ 9–12: เริ่ม review code คนอื่น / ลด hand-holding
- สัปดาห์ 13–16: ทำงานอิสระมากขึ้น, เตรียมส่งต่อความรู้ให้รุ่นถัดไป

### 4. วิธีวัดผล
- ตัวอย่าง KPI ที่ "วัดได้จริง": จำนวน PR merged, เวลาเฉลี่ยตอบ code review, จำนวนคำถามที่ต้องถามซ้ำ, ผล 1:1 feedback
- ต้องมี checkpoint ชัดที่สัปดาห์ 4 / 8 / 12 — ผ่าน/ไม่ผ่านคืออะไร

### 5. จัดการ Low Performer
- นึกถึงสถานการณ์เฉพาะเจาะจง เช่น "ถ้า trainee ส่งงานไม่ทันตลอด" หรือ "เขียนโค้ดได้แต่ไม่กล้าถาม"
- โครง: หา root cause ก่อน (1:1 ตรง ๆ) → ตั้งเป้าหมายเล็กที่ทำได้จริง → ให้เวลา/feedback ถี่ขึ้น → checkpoint ชัดว่าถ้ายังไม่ดีขึ้นจะทำยังไง (ไม่ใช่ไล่ออกทันที แต่ก็ไม่ปล่อยลอย)
- ต้องมี empathy แต่ realistic — ไม่ใช่ "ทุกคนดีขึ้นได้เสมอ"

### 6. Retention & Vision
- ทำไมคนถึงอยากอยู่ต่อ (growth path, autonomy, การถูกมองเห็นผลงาน)
- มองภาพ 3–5 ปี: trainee คนนี้อาจโตเป็นอะไร (senior, mentor รุ่นถัดไป)

### 7. Plan B
- ถ้า retain ได้แค่ 2/5 จะเรียนรู้อะไรจากมัน จะปรับ process ยังไงรอบหน้า

## Red Flags ที่ต้องเลี่ยง (จาก assignment-spec)
- เขียนแบบ "ตำรา HR" ลอย ๆ ไม่มีมุมมองตัวเอง
- ไม่มีตัวอย่างสถานการณ์ concrete
- ไม่มี measurable goal ที่จับต้องได้
