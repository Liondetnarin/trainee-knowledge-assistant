Part 4 · Legacy System Thinking · 20 คะแนน

เริ่มยังไงกับระบบเก่า 10 ปีที่ไม่มีใครเข้าใจ

Scenario: เว็บลูกค้าอายุ 10 ปี ทีม dev เดิมลาออกหมด ไม่มีเอกสาร ไม่มี test - ลูกค้ามี 3 เรื่อง: เว็บช้า, เจอไฟล์แปลกใน server, อยาก deploy feature ใหม่แต่ไม่มีใครกล้า · มีเวลา 2 สัปดาห์ทำ discovery · ต้นฉบับ: docs/part-4-legacy.md

คำถามที่ 1: 3 วันแรกจะทำอะไร

ลำดับ priority

ผมไม่เคยเจอ legacy system แบบนี้ตรง ๆ มาก่อนครับ เคยแต่นึกภาพว่าถ้าเจอจะเป็นยังไง เพราะงั้นแนวทางของผมจะเน้นความระมัดระวังเป็นหลัก ไม่รีบ - จำแนกก่อนเสมอว่าอะไรเสี่ยงกระทบ prod มาก/น้อย ก่อนตัดสินใจลงมือ

Day 1–3

วันสิ่งที่ทำ
Day 1ศึกษาข้อมูลก่อน - อ่าน docs ของงานที่ใกล้เคียงกันเท่าที่มี, จำแนกว่าอะไรกระทบ prod เสี่ยงมาก/น้อย พร้อม backup ข้อมูลไว้ก่อนเสมอ เพื่อไม่ให้เกิดความเสี่ยงต่อข้อมูลและผู้ใช้งานที่ใช้อยู่จริง
Day 2ใช้ AI ช่วยอ่าน/สรุปว่าระบบคืออะไร มีอะไรที่จับต้องได้โดยไม่เสี่ยง ไม่กระทบ prod หรือ user บ้าง พร้อมจดบันทึกเป็น step อย่างมีระบบ เพื่อให้ทำงานต่อไปได้ถูกทาง
Day 3พอรู้ความเสี่ยงและ backup แล้ว เริ่ม debug ปัญหาจริง (เช่นความช้า) รายงานผลเพื่อหาแนวทางแก้ไขที่ตรงกับที่ลูกค้าต้องการ

อะไรที่บ่งชี้ว่าเว็บช้าเพราะอะไรได้บ้าง"

เพื่อไม่ให้เกิดผลกระทบต่อลูกค้าที่ใช้งานอยู่ใน server"

ถ้าไม่รู้ จะถามใคร

พี่ในทีมที่ส่งงานมาให้ - เรื่อง context ธุรกิจและประวัติปัญหาเก่า ๆ, sysadmin ที่ดูแล infra - เรื่องสิทธิ์เข้าถึงและ deploy process เดิม, และถ้ายังไม่ชัดจะลองถามกลุ่ม/platform ที่เจอเหตุการณ์ใกล้เคียงกันมาก่อน ควบคู่กับถาม AI

เรื่องที่กังวล

กลัวว่าแค่สำรวจเฉย ๆ ก็ดันไป trigger อะไรที่กระทบ prod เพราะไม่เคยเจอระบบแบบนี้ตรง ๆ มาก่อน เลยอยากทำทุกอย่างที่เป็น read-only ให้มากที่สุดในช่วงแรก ค่อยขยับไปทำอะไรที่เสี่ยงขึ้นทีหลัง

จบ 3 วัน deliver อะไร

Discovery doc สั้น ๆ (architecture เท่าที่เข้าใจ, root cause เบื้องต้นของความช้า, risk list, คำถามที่ยังไม่มีคำตอบ) พร้อมเสนอลำดับความสำคัญของงานสัปดาห์ถัดไป - ถ้า 3 วันไม่พอให้สรุปชัด จะขอเวลาเพิ่มอีก 2-3 วันแทนที่จะรีบสรุปแบบเดา ๆ

คำถามที่ 2: ไฟล์ .php obfuscate ใน folder images/

คิดว่าคืออะไร

ไม่แน่ใจเลยจริง ๆ ครับ ไฟล์ .php ที่อยู่ใน images/ ปกติมักเป็นไฟล์สำหรับจัดการรูปภาพ (อัปโหลด/resize) แต่ก็อาจเป็นโค้ดที่แฮกเกอร์ฝังไว้ (malware) ถ้าไม่ใช่คนที่ทีมสร้างขึ้นเอง อาจเป็นได้ทั้งสองแบบ ยังฟันธงไม่ได้ด้วยข้อมูลเท่าที่มีตอนนี้

คำถามก่อนตัดสินใจ

  • มี log ไหมว่าไฟล์นี้เคยถูกเรียกใช้งานจริงบ้าง ล่าสุดเมื่อไหร่
  • ฟีเจอร์ในระบบ (เช่น upload รูป) เคยเรียกใช้ไฟล์นี้จริงหรือเปล่า

จะทำอย่างไร (ลบ / เก็บ / อื่น ๆ)

ยังไม่ลบครับ - ถ้าเป็นไฟล์จัดการรูปภาพจริง การลบอาจกระทบ prod (ฟีเจอร์พัง) แต่ถ้าเป็น malware การลบทันทีจะเสียหลักฐานที่ใช้สืบต่อได้ ดังนั้นจะ quarantine ไฟล์ไว้ก่อน (ย้ายออกจาก path ที่ execute ได้ หรือ backup แยกต่างหาก) และไม่รันไฟล์นี้บน prod เด็ดขาดไม่ว่าจะเป็นแบบไหน

อะไรต้องเช็คเพิ่ม

  • Server log ว่ามีการเข้าถึงไฟล์นี้จาก IP แปลก ๆ ไหม
  • ไฟล์อื่นในโฟลเดอร์เดียวกันที่ modify time ใกล้เคียงกัน
  • สิทธิ์ของโฟลเดอร์ images/ ว่าอนุญาตให้ execute PHP ได้จริงไหม (ถ้าใช่ นั่นคือสัญญาณอันตราย)

ใครควรรู้

พี่ในทีมที่ส่งงานมาให้ก่อน แล้วถ้าข้อมูลชี้ไปทางเรื่องความปลอดภัยจริง ต้องแจ้งขึ้นไปอีกระดับ (security/lead) และอาจต้องแจ้งลูกค้าด้วย เพราะกระทบความปลอดภัยของระบบที่ยังมีคนใช้งานอยู่จริง

Session 1 - checklist ก่อนแตะ production
Prompt: "ช่วย list checklist สิ่งที่ควรเช็คก่อนแตะ production server ที่ไม่มีเอกสารเลย"
AI ตอบ: แนะนำให้ backup/snapshot, เช็ค log, เช็คสิทธิ์เข้าถึงก่อนเสมอ
ปรับเอง: เพิ่มขั้นตอนจำแนกความเสี่ยงมาก/น้อยก่อน backup เพราะ AI ไม่รู้ context ว่า server ไหนกระทบ user จริงมากกว่ากัน ต้องคิดเองว่าจะ priority ตัวไหนก่อน

Session 2 - ไฟล์ .php obfuscate
Prompt: "เจอไฟล์ .php obfuscate ใน images/ ควรทำยังไง"
AI ตอบ: บอกว่าอาจเป็น malware แนะนำให้ scan และไม่รันไฟล์
ปรับเอง: เพิ่มมุมว่าไฟล์ใน images/ อาจเป็นไฟล์จัดการรูปปกติด้วย ไม่ใช่ malware เสมอไป เพราะ AI ตอบเอียงไปทาง security ฝ่ายเดียว โดยไม่ได้คิดถึงความเป็นไปได้ที่จะเป็น false positive
